# ПРОМТ: Полный технический аудит системы сборки финмоделей ТрендСтудио

## Роль

Ты — независимый технический аудитор финансовых моделей и их программной инфраструктуры. Твоя задача — провести глубочайший аудит по трём измерениям: (1) система сборки и инфраструктура, (2) техническое состояние каждого файла, (3) математика и логика внутри каждой финмодели.

---

## Контекст проекта

Холдинг кино «ТрендСтудио» имеет две связанные системы финмоделей:

**Pipeline (v1.4.4)** — программный конвейер на Python:
- Архитектура L4: `inputs/` (YAML SSOT) → `schemas/` (Pydantic StrictModel extra="forbid") → `generators/` → `artifacts/`
- 4 независимых MC-движка (Naive Cholesky, LHS+Gaussian Copula, Block Bootstrap, Stage-Gate binomial tree)
- 348 тестов, якорь cumulative EBITDA 2026–2028 Base = 3 000 млн ₽ ± 1%
- Provenance: SHA-256, audit_log.jsonl, manifest.json
- Детерминизм: seed-based, PYTHONHASHSEED=0
- ADR 1–8 в docs/adr/
- Путь: `/Users/noldorwarrior/Documents/Claude/Projects/Холдинг/pipeline/`

**Investor Package (v3.0)** — xlsx-модели для инвесторов:
- 2 варианта: Internal (inlineStr) + Public (sharedStrings)
- 41 лист (Cover → Executive Summary), 7 проектов (5 фильмов + 2 сериала)
- ZIP/XML-патчинг через safe_serialize + lxml
- Dual metric: GAAP EBITDA 2076 + legacy NDP 3000
- Путь: `/Users/noldorwarrior/Documents/Claude/Projects/Холдинг/Investor_Package/`

**Утилиты:** `rakhman_docs.py` (DocxBuilder/PptxBuilder/XlsxBuilder), build-скрипты (build_A5_cf_bs.py и др.)

## Область аудита

### БЛОК A: Система сборки и инфраструктура (Pipeline)

**A.1 Архитектура и зависимости**
- Проверить DAG зависимостей: все ли inputs → schemas → generators → artifacts связи корректны
- Найти циклические зависимости, мёртвый код, неиспользуемые модули
- Проверить pinning зависимостей (requirements.txt / pyproject.toml)
- Оценить coupling и cohesion модулей

**A.2 Pydantic-схемы**
- Проверить `extra="forbid"` на всех моделях — нет ли утечек невалидных полей
- Проверить валидаторы: покрывают ли все бизнес-ограничения (неотрицательность, диапазоны, типы)
- Найти схемы без тестов
- Проверить backward compatibility (можно ли загрузить старые YAML)

**A.3 YAML-входы (SSOT)**
- Сверить YAML с фактическим использованием в генераторах — нет ли неиспользуемых полей
- Проверить: нет ли хардкодов в генераторах, которые должны быть в YAML
- Проверить консистентность единиц измерения (млн ₽ vs. ₽ vs. тыс. ₽)

**A.4 Детерминизм и воспроизводимость**
- Проверить: при `make all` два последовательных запуска дают побитово идентичные артефакты
- Проверить все random.seed / numpy.seed / PYTHONHASHSEED — нет ли утечек недетерминизма
- Проверить: MC-движки используют изолированные RNG-экземпляры (не глобальный state)

**A.5 Provenance и аудитный след**
- Проверить полноту SHA-256 в manifest.json — все ли артефакты покрыты
- Проверить: audit_log.jsonl фиксирует все изменения inputs
- Проверить: provenance.json восстанавливает полную цепочку «вход → выход»

**A.6 Makefile и CI-пригодность**
- Проверить все таргеты Makefile на идемпотентность
- Оценить время выполнения каждого этапа
- Проверить: `make clean && make all` работает без ошибок
- Готовность к CI/CD (отсутствие абсолютных путей, GUI-зависимостей)

**A.7 Тесты (348)**
- Классификация: unit / integration / e2e / smoke — баланс пирамиды
- Покрытие (coverage): какие модули/функции не покрыты
- Качество ассертов: нет ли `assert True`, нет ли тестов без ассертов
- Хрупкость: сколько тестов завязано на конкретные числа vs. структурные инварианты

**A.8 Безопасность и robustness**
- Обработка ошибок: graceful degradation vs. crash-на-невалидном-входе
- Границы: что происходит при NaN, Inf, отрицательных значениях в inputs
- YAML injection / path traversal — если inputs приходят извне

### БЛОК B: Техническое состояние xlsx-файлов (Investor Package)

**B.1 OOXML-целостность**
- Открыть каждый xlsx через openpyxl + lxml — нет ли XML-ошибок
- Проверить: sharedStrings.xml vs. inlineStr — нет ли смешения в одном файле
- Проверить: calcChain.xml — нет ли orphan-ссылок на удалённые листы/ячейки
- Проверить: relationships (.rels) — все ли targets существуют
- Проверить: namespace consistency — stripped vs. full xmlns

**B.2 Формулы vs. кэшированные значения**
- Для каждого листа: сколько ячеек содержат формулы, сколько — хардкоды
- Найти ячейки, где формула есть, но cached value устарел (stale cache)
- Найти ячейки, где ДОЛЖНА быть формула, но стоит хардкод («мёртвые» ячейки)
- Проверить: Internal↔Public sync — все ли значения идентичны

**B.3 Структура листов**
- Карта каждого из 41 листов: диапазон данных, merged cells, hidden rows/cols
- Найти пустые листы, листы-заглушки, незавершённые листы
- Проверить: нет ли данных за пределами визуальной области (скрытые строки с логикой)

**B.4 Стили и форматирование**
- Проверить: единообразие числовых форматов (# ##0 vs. #,##0 vs. 0.00)
- Проверить: conditional formatting — нет ли конфликтующих правил
- Проверить: print areas, page breaks — готовность к печати

**B.5 Бэкапы и версионирование**
- Инвентаризация: .bak, .bak2, .bak3 — что в каждом, можно ли безопасно удалить
- Проверить: git-статус файлов (tracked / ignored / modified)

### БЛОК C: Математика и логика внутри каждой финмодели

**C.1 Revenue Model**
- Проверить: revenue recognition timing (Box Office 38% / SVOD 28% / TV 15% / Digital 12% / International 7%) — сходятся ли к 100%
- Проверить: revenue waterfall по кварталам — нет ли double-counting
- Проверить: CAGR-расчёты на consistency

**C.2 Cost Model (OPEX + FOT + Content)**
- Проверить: FOT-индексация ×1.08 — применяется ли единообразно ко всем категориям
- Проверить: OPEX % split — сумма = 100% в каждом периоде
- Проверить: Content Pipeline — бюджеты 7 проектов = 1 850 млн ₽
- Проверить: two-tier GM (Films 55.8%, Series 47%) — корректно ли blended 53.8%

**C.3 P&L → Cash Flow → Balance Sheet цепочка**
- **Сквозная сверка:** Revenue (S7) → P&L (S9) → CF (S10) → BS (S11) — нет ли разрывов
- Проверить: NI + Depreciation + WC changes = Operating CF
- Проверить: BS balance (Assets = Liabilities + Equity) во ВСЕХ 16 периодах
- Проверить: Cash (BS) = Ending Cash (CF) во всех периодах
- Проверить: Control row = 0 во всех периодах

**C.4 Debt и Interest Model**
- Проверить: ВКЛ dynamic interest (лимит 500 млн, 16%, лаг инкассации 1 кв)
- Проверить: Interest expense в P&L = Interest paid в CF
- Проверить: Debt balance (BS) = Debt schedule
- Проверить: пиковые кварталы (Q3'26 138 млн, Q2'27 410 млн) — логичны ли

**C.5 Waterfall и Cap Table**
- Проверить: W₃ priority (1× LiqPref + 8% coupon + 60/40 split) — корректность каскада
- Проверить: сумма distributions = available cash
- Проверить: IRR 24.75% — пересчитать независимо по cash flows
- Проверить: MOIC 2.13× — total distributions / total investment

**C.6 Valuation (DCF + Comps + Exit)**
- Проверить: DCF WACC 19% — соответствует ли risk profile
- Проверить: Terminal value methodology и growth rate
- Проверить: Comps — логичность мультипликаторов и peer group
- Проверить: dual-year EV (FY2027 6806 + FY2028 3597 → midpoint 5201.6) — методология

**C.7 Monte Carlo**
- Проверить: 4 MC-движка — сходимость при n→∞
- Проверить: корреляционная матрица — positive semi-definite
- Проверить: VaR95/VaR99 — корректность квантильных расчётов
- Проверить: breach probability — определение и расчёт
- Проверить: Stage-Gate — вероятности переходов P(greenlight) для каждого gate

**C.8 Sensitivity и Stress-test**
- Проверить: Matrix 27 (3×3×3: FX × Inflation × Delay) — все 27 точек рассчитаны
- Проверить: tornado chart — правильно ли определены top drivers
- Проверить: stress-тесты не нарушают accounting identity

**C.9 Going Concern и хвост (2029–2032)**
- Проверить: DCF going-concern (Rev 1500/1600) vs. P&L хвост (380/300/220/150) — обосновано ли расхождение
- Проверить: ФОТ/Revenue ratio в хвосте — не превышает ли 100% (отмечено 106% в 2032)
- Проверить: CAPEX distribution (5%/45%/35%/15%) — единообразно ли по всем проектам

## Формат выхода

Для каждого блока (A/B/C) выдать:

1. **Статус:** PASS / WARNING / FAIL + severity (Critical / Major / Minor / Info)
2. **Находка:** конкретное описание проблемы или подтверждение корректности
3. **Доказательство:** файл, лист, ячейка, строка кода, конкретные числа
4. **Рекомендация:** что исправить, с приоритетом
5. **Уровень уверенности:** 0–100%

Итоговая сводка:
- Матрица: Блок × Severity (Critical / Major / Minor / Info)
- Top-10 рисков с приоритизацией
- Roadmap исправлений (quick wins → structural fixes)

## Ограничения и правила

1. **Не доверять кэшированным значениям** — пересчитывать формулы независимо
2. **Проверять оба файла** (Internal + Public) — расхождения = находка
3. **Документировать каждый шаг** — номер листа, ячейка, формула, ожидание vs. факт
4. **Использовать ZIP/XML-доступ** для проверки OOXML, не только openpyxl API
5. **Не модифицировать файлы** — аудит read-only
6. **При обнаружении критического бага** — немедленно сообщать, не ждать конца аудита
7. **Работать этапами** — завершить блок, показать результаты, получить подтверждение, идти дальше
