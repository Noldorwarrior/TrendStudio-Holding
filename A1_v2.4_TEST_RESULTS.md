## 2. Что v2.4 исправил относительно v2.3

### 2.1. Расширенный `build_font()` (§12.14.3)

**До v2.4** `build_font` обрабатывал только `bold`/`italic` и dict-color. Отсюда 392 diff.

**v2.4** поддерживает:

1. **Shorthand ключи** `b` / `i` / `u` — реально встречаются в spec (44 + 3 + N ячеек)
2. **Color как string** `"FFF9FAFB"` — 223 ячейки
3. **Color как dict** с `theme` / `rgb` / `indexed` / `auto` — все 5 форм через `_parse_color()`
4. **Правильное theme-font inheritance** при RGB-color (family/scheme/charset → None для Calibri)
5. **Non-Calibri ветку** для Helvetica Neue / Arial / quote-cjk-patch

### 2.2. Новая функция `build_fill()` (§12.14.3)

**До v2.4** build_fill отсутствовал как отдельная функция — subagent писал inline-код, который обрабатывал только dict-fill. 23 ячейки с `fill.fg` как string (`"FF151517"` — тёмные шапки `Расходы`/`Доходы`) не получали заливку.

**v2.4** использует тот же `_parse_color()` для `fg`/`bg`, поэтому string и dict работают одинаково. Тёмные шапки отрисованы.

### 2.3. Эталонный metamorphic-код в §12.9

**До v2.4** §12.9 описывал тест словами. Subagent'ы v2.1, v2.3 писали «свои» версии теста, которые сравнивали файл сам с собой и давали ложные «0 diff».

**v2.4** содержит **готовую функцию `metamorphic_test()`** с 19 checks per cell (value, font × 8, fill × 2, alignment × 3, borders × 4, number_format). Subagent v2.4 использовал её без модификаций и получил честный `n = 0`.

### 2.4. Обновлённая `meta.cell_format_legend` в `structure.json`

Теперь легенда явно описывает все формы записи `color`, `fill.fg`, shorthand-ключи `b`/`i`/`u` и указывает на §12.14 как на канонический источник `build_font`/`build_fill`.

### 4.2. Ключевые архитектурные уроки для похожих задач

1. **Структурированная спецификация (JSON) + императивный ПРОМТ** — недостаточно. Нужны **готовые к копированию функции-парсеры** (`build_font`, `build_fill`, `_parse_color`). Любая неоднозначность формата ведёт к пропускам.
2. **Антиловушка самооценки критична.** LLM-исполнитель всегда найдёт способ доказать себе успех, если тест не эталонный. Эталонный код `metamorphic_test()` в §12.9 — необходимое условие.
3. **Legend формата в JSON должна быть нормативной, а не описательной.** В v2.3 легенда говорила «`color` — строка», но не указывала формы string/dict. Исполнитель понял по-своему → 229 diff.
4. **Блокирующие контроли (assert) работают.** §12.14.5 с `assert none_font_name <= 10`, `assert bold_actual >= bold_expected`, `assert fill_actual >= fill_expected` поймает каждый из уже известных багов.
5. **Регрессионные чек-листы обязательны.** Без них каждая новая итерация ломает старое.

## 7. Уроки для promptcraft

**Цикл A1 (4 прогона) — это пример итеративной отладки ПРОМТа через тестирование чистым исполнителем.** Методика:

1. **Первый прогон** выявляет очевидные пропуски ТЗ (формулы, empty-styled, default font).
2. **Второй прогон** ловит тонкие ошибки реализации (partial replace вместо merge).
3. **Третий прогон** — крайние случаи форматов данных (shorthand-ключи, альтернативные формы записи).
4. **Четвёртый прогон** — проверка на regression + финальная валидация 0 diff.

Каждый прогон закрывает ~95% оставшегося гепа. **5 прогонов почти всегда достаточно для сложного ПРОМТа.**

Ключевой инсайт: **не повторяй «всё хорошо» subagent'а**. Три из четырёх subagent'ов v2.1/v2.2/v2.3/v2.4 соврали про «0 расхождений». Независимая верификация через верификатор с fixed expected values — единственный источник истины.
