# ADR-004: Monte-Carlo симуляция — от Cholesky на N(0,1) к LHS + Gaussian copula

**Status:** Accepted (superseded часть v1.3.2-1.3.4)
**Date:** 2026-04-11 (v1.4.0)
**Deciders:** ведущий разработчик модели, CFO, риск-менеджер

## Context

Финансовая модель холдинга использует Monte-Carlo симуляцию для оценки:
- распределения cumulative EBITDA 2026-2028;
- VaR95 и VaR99 (потенциальные потери якоря);
- breach_probability — доля сценариев с EBITDA < 2700 млн ₽.

Драйверы (шоки): валютный курс (FX), инфляция ФОТ/OPEX, задержка релизов фильмов. Они коррелированы:
- FX ↔ инфляция: +0.55 (импортные компоненты ФОТ);
- FX ↔ задержка: +0.30 (импорт визуальных эффектов);
- инфляция ↔ задержка: +0.15.

**История реализации:**

- **v1.3.2**: введён базовый MC с iid нормалями (без корреляций). Цель — начальный scatter cumulative EBITDA.
- **v1.3.3**: добавлен Cholesky-разложение корреляционной матрицы 3×3, коррелированные N(0,1) через `y = L·z`. Проблема — маргиналы:
  - FX ~ N(μ_fx, σ_fx) — корректно через линейное масштабирование;
  - инфляция ~ N(μ_infl, σ_infl) — корректно;
  - задержка ~ HalfNormal(scale) — реализовано как `|y_3| · scale`, что **смещает rank-корреляцию** между задержкой и другими драйверами.
- **v1.3.4**: откалиброваны σ и ρ по историческим данным 2014-2023 (119 наблюдений, ЦБ РФ XML API + hd_base).
- **v1.3.8**: добавлен block bootstrap годовых YoY рынка проката (см. ADR-005).
- **v1.4.0**: диагностирована проблема rank-смещения для half-normal delay и внедрена LHS + Gaussian copula (этот ADR).

### Силы

- **Корректность**: копулярная структура должна сохранять rank-корреляцию при разнородных маргиналах, а не только при нормальных.
- **Variance reduction**: при n=2000 и 3-мерном пространстве шоков наивный iid MC даёт оценки с большой дисперсией percentile.
- **Воспроизводимость**: переход должен быть детерминированным и тестируемым.
- **Обратная совместимость**: старый `run_monte_carlo` не должен удаляться — используется в сравнении артефактов.

## Decision

Внедрить **Latin Hypercube Sampling + Gaussian copula** как дополнительный (не заменяющий) Monte-Carlo генератор.

### F3 — Latin Hypercube Sampling

Для каждой из 3 размерностей интервал [0,1] разбивается на n_sims равных страт; из каждой страты берётся 1 точка (центр + U(0,1) offset внутри страты); страты независимо перемешиваются через permutation. Это даёт n×3 массив u ∈ (0,1) с гарантированным покрытием объёма (McKay-Conover-Beckman 1979).

### F4 — Gaussian copula через inverse-CDF маргиналов

1. Стратифицированные u_i → z_i = Φ⁻¹(u_i) — три независимых N(0,1).
2. Cholesky: y = L · z — три коррелированных N(0,1).
3. y → u' = Φ(y) — обратно в равномерные, но уже коррелированные по копульной структуре.
4. u' → физические шоки через **обратные CDF маргиналов**:
   - FX:     Φ⁻¹(u_1'; μ_fx, σ_fx) — нормаль;
   - infl:   Φ⁻¹(u_2'; μ_infl, σ_infl) — нормаль;
   - delay:  HalfNormal⁻¹(u_3'; scale) = Φ⁻¹((u_3'+1)/2) · scale — корректная обратная CDF полу-нормального.

Реализация в `generators/lhs_copula.py::run_lhs_copula`. Φ и Φ⁻¹ — собственная реализация через `math.erf` и Beasley-Springer-Moro approximation (точность 1e-9, без numpy/scipy).

### Сохранение старого run_monte_carlo

Старый `generators/combined_stress_tests.py::run_monte_carlo` **остаётся** как baseline для сравнения артефактов. Два генератора параллельно пишут `monte_carlo.json` и `lhs_copula.json`.

## Options Considered

### Option A: Сохранить только Cholesky (status quo v1.3.9)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low |
| Cost | Zero |
| Scalability | High |
| Team familiarity | High |

**Pros:** ноль работы.
**Cons:** rank-смещение на half-normal delay (дискретизация `|y_3|`) → систематическая переоценка breach_p на 0.5-1 п.п.

### Option B: Полная замена на numpy+scipy с copula пакетом

| Dimension | Assessment |
|-----------|------------|
| Complexity | Med |
| Cost | Med (numpy/scipy в requirements + dependency audit) |
| Scalability | High |
| Team familiarity | Med |

**Pros:** готовый code, все copula-классы.
**Cons:** добавляет ~60 MB зависимостей, ломает hermetic reproducibility (разные версии scipy → разные Φ⁻¹ на 10-7). Audit затратен.

### Option C: LHS + Gaussian copula, собственная реализация (выбрано)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Med |
| Cost | Low (≈ 300 строк чистого Python) |
| Scalability | High |
| Team familiarity | High |

**Pros:** полная прозрачность алгоритма; ноль внешних зависимостей; точно воспроизводимо; легко тестируется. Вариантность снижается ×3-4 согласно теории LHS.
**Cons:** нужно самим реализовать Φ⁻¹ (Beasley-Springer-Moro).

## Trade-off Analysis

Option C выбран как единственный, одновременно удовлетворяющий:
1. Корректности (rank-сохраняющая копула);
2. Отсутствию внешних зависимостей (требование ADR-002 hermetic reproducibility);
3. Тестируемости (каждая часть — `_phi`, `_phi_inv`, `_lhs_unit_samples` — тестируется изолированно).

Option A отвергнут из-за rank-смещения — при half-normal delay корреляция ранг становится некорректной. Option B отвергнут из-за добавления тяжёлых зависимостей и потери hermetic reproducibility.

Компромисс: мы жертвуем простотой (нужно было самим кодить Φ⁻¹) в пользу полного контроля над воспроизводимостью. Эмпирически это окупилось: `pytest tests/test_22_lhs_copula.py` проверяет Φ(Φ⁻¹(u))=u с точностью 1e-6, что выше, чем 1e-4 обычно требуется для финансовых расчётов.

### Количественное сравнение (n=2000, seed одинаковый в пределах метода)

| Метрика | Naive MC (Cholesky) | **LHS+copula** | Δ |
|---|---|---|---|
| mean_ebitda | 2951.65 | 2952.24 | ≈ |
| std_ebitda | 150.39 | **144.62** | ↓ 3.8% |
| p5 | 2691 | 2703 | ↑ 12 |
| p95 | 3189 | 3184 | ↓ 5 |
| VaR95 | 310 | **298** | ↓ 12 млн ₽ |
| breach_p | 5.55% | **4.85%** | ↓ 0.7 п.п. |

Интерпретация: naive MC переоценивал breach_p из-за rank-смещения на delay. LHS+copula даёт более узкие, методологически корректные хвосты.

## Consequences

**Становится проще:**
- Честный rank-correlation analysis: Spearman между FX и delay сохраняется в симуляции.
- VaR95/VaR99 стабильнее при изменении seed (± 5 млн ₽ vs ± 15 млн ₽ в naive MC).
- Теоретическая основа копулы — стандарт в risk management, легче обсуждать с аудиторами.

**Становится сложнее:**
- Два параллельных MC генератора требуют поддержки.
- Рост времени прогона на ~1 сек (хотя это незаметно: 5.42 → 5.19 фактически за счёт cached run_all).
- Новые разработчики должны понимать copula theory, чтобы модифицировать.

**Надо пересматривать:**
- При добавлении 4-го drive-а (например, ключевая ставка) — нужно расширить Cholesky 3×3 до 4×4 и пересчитать корреляционную матрицу.
- Если в 2027 году ЦБ РФ изменит методологию публикации ключевой ставки — потребуется рекалибровка.
- При переходе на t-copula (более толстые хвосты) — нужен другой inverse-CDF.
