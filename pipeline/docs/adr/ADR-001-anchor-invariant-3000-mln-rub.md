# ADR-001: Якорный инвариант cumulative EBITDA 2026-2028 = 3000 млн ₽ ± 1%

**Status:** Accepted
**Date:** 2026-02-15 (зафиксирован в v1.3.0)
**Deciders:** CFO Холдинга «ТрендСтудио», руководитель финмодели, продюсерский блок

## Context

Финансовая модель холдинга «ТрендСтудио» должна быть одновременно:

1. **Верифицируемой** — любое изменение входных данных, формул или сценариев должно быть регистрируемо, а итоговая метрика — проверяема против одного общедоступного числа;
2. **Согласованной** с KPI, уже утверждёнными советом директоров (рост EBITDA до 3 млрд ₽ накопленно за трёхлетку 2026-2028);
3. **Устойчивой к дрейфу** — при 14+ YAML входах и 15+ генераторах цепочка расчётов может незаметно «уплыть» при любой мелкой правке.

Отсутствие якорного инварианта приводит к тому, что итоговая цифра становится «плавающей» — каждый коммит сдвигает её на 5-20 млн ₽, и никто не может ответить на вопрос «а базовая цифра правильная?».

### Силы

- **Цель бизнеса**: совет утвердил Base = 3 млрд ₽ кумулятивно 2026-2028. Это публичное обязательство.
- **Точность модели**: реалистичные допущения дают ±1% волатильность цифры при изменении отдельных коэффициентов на второй знак после запятой.
- **Надёжность**: отсутствие регрессий критично — модель читают внешние аудиторы и инвесторы.

## Decision

Зафиксировать **cumulative_ebitda_base_2026_2028 = 3000 млн ₽ ± 1%** как якорный инвариант пайплайна и проверять его в каждом тестовом прогоне.

Границы инварианта: **[2970; 3030] млн ₽** (±1% = ±30 млн ₽).

Реализация:
- Схема `schemas/anchor.py::Anchor` хранит target=3000 и tolerance=0.01.
- Конфиг в `inputs/macro.yaml` под секцией `anchor:`.
- Проверка в `tests/test_03_anchor_invariant.py::test_base_cumulative_ebitda_hits_anchor`.
- CI-gate: любой прогон `run_pipeline.py` печатает `якорь: actual=X δ=±Y% [PASS/FAIL]`.
- **FAIL блокирует релиз** — pytest падает, bundle не создаётся.

## Options Considered

### Option A: Жёсткий инвариант ±0.5%

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low |
| Cost | Low |
| Scalability | High |
| Team familiarity | High |

**Pros:** максимальная чувствительность к регрессиям.
**Cons:** ложные срабатывания при допустимых уточнениях коэффициентов (например, округление НДС на 2-й знак). Каждые 2-3 коммита потребуют пересчёта инварианта.

### Option B: Мягкий инвариант ±1% (выбрано)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low |
| Cost | Low |
| Scalability | High |
| Team familiarity | High |

**Pros:** ловит реальные баги (смена знака, потеря сегмента, ошибка в масштабе), но не мешает тонким калибровкам. 30 млн ₽ ≈ 2 средних фильма = естественная бизнес-единица.
**Cons:** теоретически пропускает медленный дрейф < 0.5% за релиз.

### Option C: Многоуровневый (base + cons + opt инварианты)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Med |
| Cost | Med |
| Scalability | Med |
| Team familiarity | Med |

**Pros:** ловит изменения в сценарном ranging.
**Cons:** увеличивает maintenance — каждое обоснованное расширение hit-rate требует обновления 3 инвариантов. Нет KPI от совета для cons/opt.

## Trade-off Analysis

Option B выбран как компромисс между чувствительностью (A) и простотой обслуживания. Граница ±1% совпадает с:
- Типичной ошибкой округления налогов (НДС 20%, налог на прибыль 25%) на горизонте 3 лет.
- Минимальной «замечаемой» суммой для CFO (≥ 30 млн ₽).
- Пределом, при котором разложение по драйверам остаётся корректным (если δ > 1%, то один из коэффициентов точно сломан).

Option A отвергнут из-за риска блокировки релизов на безобидных правках. Option C отвергнут как преждевременная оптимизация: когда появится KPI по cons/opt — вернёмся к этому вопросу.

## Consequences

**Становится проще:**
- Любой коммит проверяется за 300 мс через pytest test_03.
- Код-ревью сводится к вопросу «якорь не сломан?» — если PASS, структурные изменения допустимы.
- Внешние аудиторы получают одно число для быстрой верификации.

**Становится сложнее:**
- Любое обоснованное изменение Base KPI (например, рост на 10%) требует явного обновления `inputs/macro.yaml::anchor.target` с обоснованием в commit message.
- Инвариант привязывает модель к конкретному слейту и сценарию — при смене слейта в 2029 потребуется новый инвариант.

**Надо пересматривать:**
- При утверждении нового стратегического плана на 2029-2031 — обновить target и tolerance.
- При изменении налогового режима (ставка на прибыль) — пересчитать границы.
