# ADR-002: Архитектура L4+N3 — разделение inputs/schemas/generators/artifacts

**Status:** Accepted
**Date:** 2026-02-20 (v1.3.1)
**Deciders:** ведущий разработчик модели, CFO

## Context

К версии v1.2 финансовая модель холдинга разрослась до ~8000 строк Python-кода с 80+ допущениями, перемешанными с расчётами. Проявились проблемы:

1. **Магические числа в коде** — коэффициенты hit-rate, доли слейта, налоговые ставки разбросаны по 15 файлам.
2. **Отсутствие контрактов** — любой опечаток в словаре ломал расчёт без понятной ошибки.
3. **Невоспроизводимость** — два запуска на одних данных давали разные цифры из-за несогласованного seed.
4. **Невозможность аудита** — аудитор не мог сказать «эта цифра пришла отсюда».

### Силы

- **Контроль версий**: любая цифра должна быть текстовой — чтобы git diff показывал смысловые изменения.
- **Верифицируемость**: каждый input должен иметь явную pydantic-схему.
- **Тестируемость**: генераторы должны быть чистыми функциями без состояния.
- **Traceability**: каждый артефакт должен быть привязан к конкретным входам через provenance.

## Decision

Разделить пайплайн на **4 слоя** (L4) и **3 артефакта provenance** (N3):

### L4 — 4 слоя кодовой базы

```
L1: inputs/        ← 18 YAML файлов (ВСЕ допущения и коэффициенты)
L2: schemas/       ← Pydantic StrictModel контракты для каждого inputs/*.yaml
L3: generators/    ← чистые функции: ValidatedInputs → ModelResult
L4: artifacts/     ← xlsx, docx, json, pptx (derived)
```

Правила:
- **L1 → L2**: inputs загружаются через `schemas/inputs.py::load_inputs()`. Загрузка падает при любой опечатке.
- **L2 → L3**: все генераторы принимают `ValidatedInputs` как единственный источник истины. Никаких globals, никаких словарных захардкоженных значений.
- **L3 → L4**: артефакты строятся из результатов генераторов без повторного чтения inputs. Это гарантирует консистентность между моделью и отчётом.

### N3 — 3 артефакта воспроизводимости

```
logs/provenance.json    ← что использовалось: SHA-256 каждого input, git rev, seed, время
logs/manifest.json      ← какие артефакты получены: SHA-256 каждого output файла
docs/adr/               ← почему так решено (этот каталог)
```

## Options Considered

### Option A: Монолитный модуль (status quo v1.2)

| Dimension | Assessment |
|-----------|------------|
| Complexity | High (для новых разработчиков) |
| Cost | Low (ничего не менять) |
| Scalability | Low (линейный рост хаоса) |
| Team familiarity | High (все привыкли) |

**Pros:** нулевой cost миграции.
**Cons:** неаудируемо, нет контрактов, никакой воспроизводимости.

### Option B: L4+N3 (выбрано)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Med (ясные границы слоёв) |
| Cost | Med (миграция всех допущений в YAML) |
| Scalability | High (линейное расширение: добавить YAML → schema → generator) |
| Team familiarity | Med (pydantic новый для большинства) |

**Pros:** полный аудит трейл, тесты, CI-gate, provenance.
**Cons:** обязательность писать схему под каждый новый input.

### Option C: Event-sourced архитектура с immutable inputs

| Dimension | Assessment |
|-----------|------------|
| Complexity | Very High |
| Cost | High (нужен event store) |
| Scalability | High (time-travel аудит) |
| Team familiarity | Low |

**Pros:** возможность time-travel на любую точку.
**Cons:** overkill для финмодели с релизом раз в квартал. Event store добавляет 2-3 новых зависимости.

## Trade-off Analysis

Option B выбран как минимальная достаточная архитектура. L4 даёт ясные границы слоёв, N3 — воспроизводимость. Option A отвергнут из-за фундаментальной неаудируемости. Option C — преждевременная оптимизация: time-travel нужен <5% запросов, а event store — ~500 строк кода и новая зависимость.

Ключевой компромисс: мы жертвуем скоростью первой реализации (обязательность pydantic-схем для каждого input) в пользу долгосрочной поддерживаемости. После v1.3 этот выбор окупился: ни одной регрессии якоря за 15+ релизов.

## Consequences

**Становится проще:**
- Новый generator добавляется за один коммит: schema → YAML → generator → test.
- Аудитор получает `provenance.json` и видит точный граф зависимостей.
- Семантический diff: `git diff inputs/` показывает бизнес-смысл, а не технические правки.

**Становится сложнее:**
- Любой новый источник данных требует отдельной pydantic-схемы (mypy-строгая).
- Refactoring формулы требует одновременного обновления schema, generator, test — 3 файла минимум.
- Кривая обучения для новых разработчиков выше (pydantic StrictModel).

**Надо пересматривать:**
- При достижении 30+ YAML файлов — возможно, потребуется иерархический loader с namespacing.
- При переходе на distributed computing — генераторы должны остаться чистыми функциями.
