# ПРОМТ: Полный аудит системы сборки финмоделей — v1 (драфт)

---

## Роль

Ты — старший финансовый инженер и OOXML-специалист. Твоя задача — провести исчерпывающий аудит системы генерации инвестиционных финансовых моделей (xlsx) и самих файлов-артефактов. Аудит должен покрыть три слоя: **техническая инфраструктура** (Python-генератор, OOXML-структура), **математическая корректность** (формулы, балансы, каскады расчётов) и **логическая связность** (межлистовые зависимости, бизнес-правила, водопад распределения).

## Контекст

### Система сборки
- **Генератор**: `build_A5_cf_bs.py` — Python-скрипт, создающий листы `10_Cash_Flow` и `11_Balance_Sheet` в каждом xlsx-файле
- **Waterfall**: W₃ — трёхступенчатый приоритет (Stage 1: 1× Liq Pref 1250 → инвестору, Stage 2: 8% купон 500 → инвестору, Stage 3: 60/40 residual)
- **Технология**: ZIP-level XML surgery (lxml + zipfile) — прямая запись в OOXML без openpyxl-генерации

## Задача

Провести **полный трёхслойный аудит** по следующей структуре:

### СЛОЙ 1: Техническая инфраструктура

**1.1. Аудит генератора `build_A5_cf_bs.py`**
- Архитектура кода: структура функций, точки входа, потоки данных
- Обработка ошибок: есть ли try/except, валидация входных данных, граничные случаи
- Детерминизм: гарантирует ли скрипт одинаковый результат при повторном запуске
- Зависимости: какие библиотеки используются, есть ли requirements.txt
- Покрытие тестами: есть ли тесты, что покрыто, что нет

**1.2. Аудит OOXML-структуры каждого xlsx**
- Целостность ZIP-архива: все ли XML-части на месте, нет ли сиротских связей
- Content_Types.xml: все ли типы зарегистрированы
- Relationships: нет ли битых ссылок
- Styles.xml: количество стилей, дубликаты, неиспользуемые записи
- SharedStrings: наличие/отсутствие, корректность
- CalcChain: наличие/отсутствие, влияние на пересчёт
- Пустые стилизованные ячейки: количество, процент от общего числа ячеек
- Размер файла и степень сжатия

### СЛОЙ 2: Математическая корректность

**2.1. Баланс (Balance Sheet — лист 11)**
- Тождество: Assets (A4) ≡ Liabilities + Equity (B4) для КАЖДОГО периода (столбцы D–S)
- Control row: C1 = A4 − B4 = 0 для всех периодов
- Компоненты A: A1 (Cash) + A2 (Pipeline) + A3 (Receivables) = A4
- Компоненты B: B1 (Loan) + B2 (Producer Equity) + B3 (Retained Earnings) = B4
- Допускаются ли отрицательные B1/B2? (Да — это корректное поведение при W₃)

**2.2. Денежный поток (Cash Flow — лист 10)**
- Opening Cash (row 1) + Net Cash Flow = Ending Cash (row 23)
- Ending Cash периода N = Opening Cash периода N+1
- Суммы по разделам: Operating + Investing + Financing = Net Cash Flow
- Distribution split: сумма distributions to investor + to producer = total distributions

**2.3. P&L (лист 09)**
- Revenue − COGS = Gross Profit
- Gross Profit − OpEx = EBITDA
- EBITDA − D&A − Interest − Tax = Net Income
- Сходимость Net Income → Retained Earnings в BS

**2.4. Cross-sheet Cash Sync**
- BS A1 (Cash) = CF row 23 (Ending Cash) для КАЖДОГО периода
- BS B3 (Retained Earnings) = cumulative P&L Net Income

**2.5. Водопад распределения (W₃)**
- Stage 1: первые 1250 → инвестору (1× Liq Pref)
- Stage 2: следующие 500 → инвестору (8% купон)
- Stage 3: остаток × 60% → инвестору, × 40% → продюсеру
- Кумулятивная проверка: сумма всех распределений = NDP
- Проверка по периодам: в каждом году waterfall применяется к периодическому распределению с учётом оставшихся лимитов Stage 1/2

### СЛОЙ 3: Логическая связность

**3.1. Межлистовые зависимости**
- Pipeline (листы 01–08) → P&L → Cash Flow → Balance Sheet — цепочка не разорвана
- Summary (лист 01) корректно агрегирует данные из всех нижестоящих листов
- IRR/NPV на Summary соответствуют денежным потокам на Cash Flow

**3.2. Internal vs Public**
- Internal содержит все данные, Public — маскированные (какие именно поля скрыты?)
- Все нескрытые данные в Public идентичны Internal
- Нет ли утечки Internal-данных в Public-версию

**3.3. v1.0 vs v3 — структурная когерентность**
- Одинаковая структура листов (42 листа в каждом)
- Одинаковая структура строк/столбцов в аналитических листах
- Различия ТОЛЬКО в данных (12→7 проектов) и производных расчётах

**3.4. Бизнес-правила**
- Бюджет проекта: v1 = 378.75 млн/проект, v3 = 649.29 млн/проект — корректно?
- Якорный доход: 3000 млн ₽ — одинаков в обеих версиях?
- Горизонт модели: 2025–2032 (16 периодов: 8 полугодий или 8 лет?)

## Формат вывода

Для каждого слоя:
1. **Таблица проверок** — чеклист с результатами ✓/✗/⚠ по каждому файлу
2. **Детализация дефектов** — для каждого ✗/⚠: что именно не так, где, какое значение ожидалось vs фактическое
3. **Severity** — Critical / Major / Minor / Info
4. **Рекомендации** — конкретные действия для исправления

Финальный блок:
- **Сводная матрица** 4×N (файл × проверка) с цветовой кодировкой
- **Общая оценка** готовности каждого файла к использованию (шкала 0–100%)
- **Приоритезированный backlog** дефектов для исправления

## Ограничения
- Не модифицировать файлы — только аудит и отчёт
- Все проверки с цифрами выполнять программно (Python) — не на глаз
- Расхождения фиксировать с точностью до ячейки (лист, строка, столбец, значение)
- Если проверка невозможна (нет данных) — зафиксировать как N/A с объяснением
